### BUG-001 тАФ `jwt.decode()` used instead of `jwt.verify()` тАФ tokens never validated

| Field | Detail |
|-------|--------|
| **Category** | Security |
| **Area** | API Authentication |
| **File** | `src/helper/getDataFromToken.ts` |
| **Line** | All usages |

**Description:**
The helper function `getDataFromToken()` calls `jwt.decode()` instead of `jwt.verify()`, meaning the token signature is never cryptographically checked. Any attacker can forge an arbitrary JWT payload with any `userId` or `role` and every protected API route will accept it as authentic. This affects 125+ API routes that rely on this helper for identity.

**Fix:**
Replace `jwt.decode(token)` with `jwt.verify(token, process.env.TOKEN_SECRET!)` wrapped in try/catch to reject tampered or expired tokens. Audit every API route that calls `getDataFromToken` to confirm the fix propagates.

| Field | Detail |
|-------|--------|
| **Category** | Security |
| **Area** | Middleware |
| **File** | `src/middleware.ts` |

**Description:**

**Description:**
The login route constructs a MongoDB `$regex` query directly from user-supplied email input without escaping special characters. An attacker can inject regex metacharacters (e.g., `.*`) to match unintended accounts or cause catastrophic backtracking (ReDoS), bypassing intended authentication logic.

### BUG-020 тАФ Magic link token design flaw тАФ bcrypt hash sent in URL and stored in DB

| Field | Detail |
|-------|--------|
| **Category** | Security |
| **Area** | Authentication |
| **File** | `src/helper/mailer.ts` |
| **Line** | 67тАУ81 |

**Description:**
The magic link flow stores a bcrypt hash of the reset token in the database and sends the raw hash in the URL. Anyone with database read access obtains the token directly and can use it for immediate account takeover.

**Fix:**
Generate `crypto.randomBytes(32).toString('hex')`, store its SHA-256 hash in the DB, send the raw token in the email URL. On verification, hash the incoming token and compare to the stored hash.
